# Autonomous Article Engine v4.6

A portable, project-agnostic article generation engine. Generates complete standalone HTML article pages from minimal input by automatically adapting to the current project's design system, detecting reusable shells and components, and producing publication-ready content with research, images, TOC, trust elements, and section-level editing capabilities.

Works on any website: company sites, blogs, editorial platforms, portfolios, agency sites, educational pages, or custom projects.

## CRITICAL RULES

### Domain Integrity
**The topic determines the article domain. The website determines the presentation.**
- "Manchester United" on a logistics site → football article, logistics site's visual style
- "Quantum Computing" on a bakery blog → physics article, bakery blog's visual style
- NEVER force a topic into the website's industry
- The website is the PUBLISHER, not the SUBJECT

### Project Adaptation
**The skill always works against the CURRENT ACTIVE PROJECT.**
- Detect the project's shell, layout, typography, colors, and components automatically
- Never assume a specific brand, site structure, or design system
- Adapt to whatever the current project provides

## OPERATION MODES

### Mode 1: Full Article Generation (default)
Runs the complete 20-step pipeline to produce a new article.

### Mode 2: Section Edit
Triggered when input matches a section edit prompt pattern:
- Contains "update section" or references a section ID
- Contains a structured edit prompt generated by the edit UI
- Runs a focused subset of the pipeline targeting only the specified section

**Section Edit Detection:**
If the input contains `SECTION_EDIT:` or references a `data-section-id`, enter Section Edit Mode (see Step 20).

- **Gemini Capability Detector** — runs the layered detection strategy: skill config override → alias matching against available tools → project-local .mcp.json → user-global config → graceful fallback. Reports which detection path succeeded.

- **Gemini Runtime Validator** — validates that the resolved Gemini tool actually works before generating images (test call or capability probe)
- **Portable Image Generation Orchestrator** — manages image generation through the normalized interface (never calls raw tool names). Copies images from MCP output to project `images/` folder using dynamic path parsing.
- **Image Generation Fallback Handler** — handles graceful degradation when ALL resolution strategies fail (placeholder HTML + exported prompts). Only activates after the full resolver chain has been exhausted.

### Structural Component Memory Modules (v4.3)
- **Structural Component Registry** — internal registry of 20+ normalized component blueprints extracted from article-components.html. Stores STRUCTURE ONLY (layout, slots, hierarchy, relationships) — no brand colors, fonts, or visual identity. Located at `config/structural-component-registry.md`.
- **Article Shell Registry** — internal blueprint for the article page shell structure (two-column layout, hero placement, sidebar, TOC). Stored within the structural component registry.
- **Blueprint-to-Project Mapper** — maps internal structural blueprints to the active project's visual language at runtime. Takes a blueprint ID + project design tokens → outputs styled HTML.
- **Runtime Visual Adapter** — applies the active project's design tokens (colors, fonts, spacing, radii, shadows) on top of structural blueprints. Never uses frozen brand values from the source project.
- **Topic-Aware Composition Selector** — selects the optimal component mix based on topic type (data-heavy, how-to, narrative, comparison, etc.) using the composition guide in the registry.

## STRUCTURAL COMPONENT MEMORY

The skill carries an **internal structural component registry** (`config/structural-component-registry.md`) containing 20+ normalized component blueprints plus article shell and hero patterns.

### Key Properties
- **Structure only** — no brand colors, fonts, or visual identity stored
- **Portable** — works in any project without external files
- **Runtime-adapted** — visual styling applied from the active project's design tokens
- **Topic-aware** — component composition varies by topic type

### Source
One-time ingestion from `article-components.html`. The registry is now self-contained within the skill. The original file is NOT required at runtime.

### Blueprint IDs Available
`bp-article-shell`, `bp-hero`, `bp-article-prose`, `bp-stats-cards`, `bp-comparison-table`, `bp-step-process`, `bp-pull-quote`, `bp-data-table`, `bp-before-after`, `bp-highlight-callout`, `bp-key-takeaways`, `bp-checklist`, `bp-timeline`, `bp-section-heading`, `bp-numbered-list`, `bp-callout`, `bp-image-caption`, `bp-faq-accordion`, `bp-feature-grid`, `bp-problem-solution`, `bp-inline-cta`, `bp-two-col-text`, `bp-mini-cards`

### Runtime Flow
1. Article Architect reads the structural registry and selects blueprints for each section
2. Draft Writer reads the selected blueprints and generates HTML structure from them
3. Runtime Visual Adapter applies the active project's design tokens to the generated HTML
4. Result: article uses proven structural patterns with project-appropriate visual styling

## STEP 13 — GEMINI CAPABILITY CHECK (inline, portable)

Before generating images, run a portable capability check using the **Gemini Tool Resolver**. This step makes Gemini integration work in ANY project, not just the one where the MCP server was originally configured.

**CRITICAL: Do NOT hardcode a single tool name.** Use the Gemini Tool Resolver to flexibly detect image generation capability.

**Known Gemini Image Tool Names (checked in order):**

The resolver checks ALL of these against the available tools at runtime:

```
1. mcp__gemini__gemini-generate-image        (standard Gemini MCP)
2. mcp__gemini__generate-image               (alternate naming)
3. mcp__gemini__generate_image               (underscore variant)
4. mcp__gemini__image-generation             (alternate naming)
5. mcp__gemini__create-image                 (alternate naming)
6. Any tool matching pattern: mcp__*gemini*__*image*  (namespace wildcard)
7. Any tool matching pattern: mcp__*gemini*__*generat*  (verb wildcard)
8. Any custom alias from engine-config.md gemini_tool_resolution.allowed_image_tool_aliases
```

**Resolution Strategy (checked in order):**

1. **Skill config override** — if `gemini_tool_resolution.preferred_image_tool_name` is set in engine-config.md, try that exact name first
2. **Known alias matching** — scan all available tools at runtime against the known names list above. If ANY match, Gemini is available. This is the primary portable method — it works regardless of which MCP package or naming convention is used.
3. **Custom alias matching** — check `gemini_tool_resolution.allowed_image_tool_aliases` from engine-config.md for user-configured aliases
4. **Namespace pattern matching** — scan available tools for any tool whose name contains both "gemini" and "image" (case-insensitive)
5. **Project-local .mcp.json** — check `{current_working_directory}/.mcp.json` for a gemini server config (confirms intent even if tool name differs)
6. **User-global config** — check `~/.claude/.mcp.json` or `~/.claude/settings.json` for global MCP server definitions
7. **Graceful fallback** — if none of the above resolve, skip image generation safely

**Capability Check Process:**

```
1. Load known aliases + any custom aliases from engine-config.md
2. Scan available tools against ALL known aliases and patterns
3. If ANY match found:
   a. Record the resolved tool name (the one that actually matched)
   b. Record the detection method (known-alias / custom-alias / pattern-match / config-override)
   c. Set gemini_available = true
   d. Set resolved_image_tool = [matched tool name]
   e. Proceed to Step 14 using resolved_image_tool
4. If NO match found:
   a. Check project-local .mcp.json for gemini server config → report if configured but tool not loaded
   b. Check user-global config → report if configured but tool not loaded
   c. Report which detection steps were attempted and why each failed
   d. Set gemini_available = false
   e. Activate fallback mode (placeholder HTML + exported prompts)
```

**IMPORTANT:** The rest of the pipeline must use `resolved_image_tool` (the name returned by the resolver), NOT a hardcoded tool name. Step 14 calls `resolved_image_tool`, not `mcp__gemini__gemini-generate-image`.

  1. Config override: [checked / skipped] — [result]
  2. Known alias scan: [checked] — [matched: tool_name / no match]

## SECTION EDIT MODE

When the skill receives a section edit prompt (from the `/apply-edit` command, direct input, or clipboard), it operates in Section Edit Mode.

### Edit Flow (Primary — Bridge Server, fully automatic)

1. User opens the generated article in a browser
2. User hovers over a section → clicks the Edit button
3. User types their desired changes in the overlay textarea
4. User clicks "Apply Edit"
5. The browser sends the SECTION_EDIT prompt to the bridge server at `http://127.0.0.1:19847/apply-edit`
6. The bridge server spawns `claude -p` to process the edit automatically
7. The section is updated — user refreshes the page to see changes

**The bridge server is auto-started after article generation.** Users can also manually start/stop it with `/start-bridge` and `/stop-bridge`.

### Edit Flow (Fallback — Clipboard + Command)

If the bridge server is not running:

1. The "Apply Edit" button auto-copies the SECTION_EDIT prompt to clipboard
2. User switches to Claude Code and types `/apply-edit`
3. The `/apply-edit` command reads the clipboard and processes the edit

If clipboard also fails, the overlay shows the raw prompt for manual copy.

### Detection

Section Edit Mode activates when input contains:
- `SECTION_EDIT:` prefix (from `/apply-edit` command or direct paste)
- A `data-section-id` reference
- "update section [id]" pattern
- A structured edit prompt generated by the page's edit UI

### Section Edit Prompt Template

The page's edit UI generates prompts in this format:

```
SECTION_EDIT:
Use the autonomous-article-engine skill to update section [SECTION_ID].

Topic: [TOPIC]
Article file: [FILENAME]
Section ID: [SECTION_ID]
Section type: [SECTION_TYPE]
Section role: [SECTION_ROLE]
Section purpose: [SECTION_PURPOSE]
Current section heading: [HEADING]

User requested change: [USER_EDIT_REQUEST]

RULES:
- Update only this section unless a minimal surrounding adjustment is required for consistency
- Preserve topic domain integrity
- Preserve page style and component compatibility
- Improve the section intelligently and professionally, not just literally
- Keep the result aligned with the rest of the article
- If the edit affects a heading, update the sidebar TOC entry to match
- Maintain the section's data attributes (data-section-id, data-section-type, data-section-role)
```

### Section Edit Execution

When in Section Edit Mode:

1. **Parse the edit prompt** — extract section ID, type, role, file path, and user instruction
2. **Read the target HTML file** — load the full article page
3. **Locate the target section** — find the section by its `data-section-id`
4. **Understand context** — read surrounding sections to understand the section's role in the article
5. **Interpret the edit request** intelligently:
   - Understand the purpose of the section
   - Understand its role in the full article
   - Interpret the user request contextually (not literally)
   - Consider tone, clarity, structure, and usefulness
6. **Rewrite the section** — produce an improved version that:
   - Addresses the user's edit request
   - Maintains domain integrity
   - Preserves component structure and CSS classes
   - Keeps `data-section-id`, `data-section-type`, `data-section-role` attributes
   - Improves quality beyond the bare minimum
7. **Check scope** — determine if neighboring sections need minimal adjustments:
   - Heading consistency with TOC
   - Factual continuity
   - Visual structure continuity
   - If TOC heading changed, update both sidebar and inline TOC entries
8. **Apply the edit** — use the Edit tool to replace the old section HTML with the new version
9. **Report the edit:**

```
SECTION EDIT COMPLETE
━━━━━━━━━━━━━━━━━━━━━━━━━
File: [filename]
Section: [section_id] ([section_type])
Change requested: [user instruction summary]
Changes made: [description of what was changed]
Scope: [section-only / section + TOC update / section + minor neighbor adjustments]
Domain integrity: preserved
━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Section Edit Intelligence Rules

The system must NOT do a weak literal rewrite. It must behave intelligently:
- Understand the purpose of the selected section
- Understand its role in the full article
- Interpret the user request contextually
- Improve clarity, structure, tone, and usefulness
- Preserve page consistency
- Keep the section aligned with the topic domain
- Keep design/component compatibility
- The update should feel like a smart editorial improvement

### Edit Scope Rules

By default, edits are section-local. Only modify surrounding sections if absolutely necessary for:
- Heading consistency
- Factual continuity
- Visual structure continuity
- TOC heading synchronization

If neighboring changes are needed, keep them minimal and explicit.

## ERROR HANDLING

- **Agent fails:** Show error, offer retry
- **Gemini MCP unavailable for research:** Degrade to WebSearch silently
- **Gemini image generation unavailable:** Run full resolver chain (known aliases → custom aliases → pattern match → config check) → only activate fallback after ALL resolution strategies fail → note in report with full diagnostic block showing each attempted step
- **Gemini image generation fails mid-run:** Retry once, then fall back to placeholder for that image. Continue with remaining images.
- **MCP output path not parseable:** Report error, skip that image, continue with others
- **Image copy to project folder fails:** Report path error, include MCP output path in report for manual copy
- **Image quality insufficient:** Regenerate weak/repetitive images (max 2 retries per image)
- **No component library found:** Switch to fallback mode automatically
- **No shell found:** Generate fallback shell automatically
- **No writing samples:** Infer style from website analysis
- **Domain drift detected:** Reject output, regenerate with stricter enforcement
- **Section edit target not found:** Report error with section ID, suggest available sections
- **User says "stop" or "cancel":** Stop immediately, report progress

## VALIDATION RULES

### A. Section ID Stability Check
Section identifiers must remain stable enough for edit reuse. IDs follow the pattern `section-{N}` where N is the section order. IDs must not change between edits unless a section is added or removed.

### B. Section Edit Scope Check
An edit should only affect the intended section unless minimal nearby edits are required. The consistency pass must verify that no unintended sections were modified.

### C. Edit Prompt Quality Check
Generated edit prompts must be structured, reusable, and clear. They must contain all required fields (section ID, type, role, topic, user instruction).

### D. Multi-Image Quality Check
All 4-6 generated images must be relevant to the topic, non-repetitive, and visually useful. No two images should have the same composition or purpose.

### E. Image-Section Match Check
Each image must fit its assigned section's purpose. A hero image should be atmospheric/broad; a contextual image should support the section's specific content.

### F. Professional UI Check
Edit controls must feel clean, integrated, and professional. They must not break the page design, appear toy-like, or distract from reading.

### G. Gemini Portability Check
No hardcoded paths to Gemini MCP output directories. All image paths must be relative to the project root. The skill must never assume a specific MCP server location or project folder.

### H. Single-Tool-Name Dependency Check
The skill must NEVER depend on one exact Gemini tool name. It must use the Gemini Tool Resolver which checks known aliases, custom aliases, pattern matching, and config overrides. If any compatible Gemini image tool is available under any recognized name, the skill must find and use it.

### I. Fallback Threshold Check
Fallback mode must ONLY activate after the full resolver chain has been exhausted (config override → known aliases → custom aliases → pattern match → project config → global config). A single tool name being absent does NOT trigger fallback — all resolution paths must fail first.

### J. Normalized Interface Check
The rest of the skill (Step 14 image generation, draft-writer, delivery report) must use `resolved_image_tool` from the Gemini Tool Resolver output. No downstream step may directly reference a hardcoded Gemini tool name.

### K. Cross-Project Capability Check
The skill must work identically in any project directory. No references to specific project paths, brand names, or folder structures outside the current working directory. Gemini resolution must be independent from where the skill was authored.

### L. Alias Resolution Check
Alternative compatible Gemini tool names must be supported. If a user's Gemini MCP exposes the tool under a different name (e.g., `mcp__gemini__generate-image` vs `mcp__gemini__gemini-generate-image`), the skill must still detect and use it.

### M. Diagnostic Output Check
The delivery report must include a Gemini integration diagnostic block showing ALL resolution steps attempted, which step succeeded (or that all failed), the resolved tool name, detection method, and generation results.

### N. Image Path Portability Check
All image `src` attributes in generated HTML must use relative paths (`images/article-{slug}-{N}.png`). Never use absolute paths or MCP output directory paths.

### O. Structure Retention Check
The internal structural registry must preserve meaningful component structures. Generated articles must use recognizable blueprint patterns from the registry.

### P. Style Stripping Check
No Axiom-specific colors (#3F7537, #D4FB50, #19342C), fonts (Alexandria, Poppins), or brand styling may be frozen into generated articles. All visual values must come from the active project's design tokens or fallback defaults.

### Q. Runtime Independence Check
The skill must NOT require `article-components.html` to be present at runtime. The internal structural registry carries all needed component structures.

### R. Blueprint Quality Check
Stored blueprints must be reusable across topics and projects. No single-use or project-specific structures.

### S. Visual Adaptation Check
Generated HTML must adapt visually to the current project's design tokens. The same blueprint used in two different projects must produce visually different (but structurally identical) output.

### T. Cross-Project Structural Reuse Check
The skill must behave identically in any project. The structural registry travels with the plugin — no external file dependencies.

### U. Section Edit UI Implementation Check (CRITICAL)
**Every generated article MUST include ALL of these — no exceptions:**
1. **Section wrappers:** Every content section wrapped in `<section>` with `id`, `class="article-section"`, and ALL five data attributes: `data-section-id`, `data-section-type`, `data-section-role`, `data-section-heading`, `data-section-purpose`
2. **Edit trigger buttons:** Every `<section class="article-section">` must contain a `<button class="section-edit-trigger">` as its first child with `data-edit-section` pointing to the section ID
3. **Edit overlay HTML:** Exactly ONE `<div class="section-edit-overlay" id="section-edit-overlay">` before closing `</body>`, containing the edit panel with textarea input, Cancel/Apply Edit buttons, status display area, fallback toggle, and prompt result display with Copy button
4. **Edit system JS:** JavaScript that wires up trigger buttons → overlay → auto-clipboard copy → status toast → fallback prompt display. Must include `ARTICLE_TOPIC` and `ARTICLE_FILE` constants, `buildEditPrompt()` helper, and `showEditStatus()` helper. The "Apply Edit" button auto-copies the SECTION_EDIT prompt to clipboard and instructs user to type `/apply-edit` in Claude Code. Fallback shows raw prompt for manual copy if clipboard fails.
5. **Edit CSS:** Full section edit system styles (trigger button, overlay, panel, input, buttons, status toast with success/error variants, fallback toggle, prompt result, copy button, print hiding)
6. **Section IDs must be numeric:** Use `section-1`, `section-2`, `section-3` etc. — following the `section-{N}` pattern for consistency with TOC anchors and the edit system JS

**If any of these are missing, the article fails validation.** The draft-writer agent's Phases F, G, H define the exact implementation. The orchestrator must verify the output contains all edit system elements before delivering.

## ADAPTATION MODE REFERENCE

| Mode | Shell | Components | When |
|------|-------|------------|------|
| **Existing** | Project's own shell | Project's own components | Project has article pages + its own component library |
| **Registry** | Project's shell (detected) | Internal structural registry blueprints + project visual tokens | Project has a shell but no article components (DEFAULT for most projects) |
| **Fallback** | Generated neutral shell | Internal structural registry blueprints + fallback design tokens | New/empty project or no usable shell patterns |

**The internal structural registry is used in both Registry and Fallback modes.** The only difference is whether the project provides a shell and visual tokens (Registry) or the skill generates/defaults them (Fallback).
